# /spec-first — 文档驱动开发周期管理

本 skill 驱动从需求到交付的完整流程：

```
需求到来 → 匹配文档 → 分析 → 更新spec → spec确认 → 开发 → 验收 → 收尾
```

适用于：新功能、Bug修复、需求调整、重构——凡是项目文档目录（默认 `docs/plans/`）下有 spec 的模块，均走此流程。

**触发后立即输出以下进度 Checklist；每个阶段完成时，重新输出当前状态的 Checklist（已完成项标为 `[x]`）：**

```
开发周期进度
- [ ] Step 0  需求接收，文档匹配
- [ ] Step 1  阶段判断
- [ ] Phase A 分析与更新 spec
- [ ] Phase B 按方案开发
- [ ] Phase C 验收
- [ ] Phase D 收尾
```

异常处理规则见 [error-handling.md](error-handling.md)。

## Step 0 — 接收需求，匹配文档

### 0.1 确认输入

检查用户是否提供了具体的需求或变更描述：

**无输入时**，输出以下提示，停止，等待用户补充，不执行后续步骤：

```
请描述您的需求或变更内容，例如：
  · 给用户列表新增按名称模糊搜索
  · 修复订单导入时出现重复数据的问题
  · 版本发布需要支持批量选择数据集
```

**有输入时**，继续执行 0.2。

### 0.2 意图分类

根据用户描述判断意图类型，决定走本 skill 还是建议用户使用其他 skill：

| 意图信号 | 意图类型 | 处理方式 |
|---|---|---|
| "分析"、"查看"、"审查"、"诊断"、"评估"、"了解"、"有哪些问题" | 只读分析 | 提示使用 `/spec-analyze`，用户确认仍要走 `/spec-first` 时继续 |
| "查找"、"搜索"、"在哪"、"有没有" | 信息检索 | 提示使用 `/spec-search`，用户确认仍要走 `/spec-first` 时继续 |
| "修改"、"修复"、"实现"、"新增"、"重构"、"调整" | 开发任务 | 继续执行 0.3 |

判断不清时，默认视为开发任务，继续执行 0.3。若描述中同时出现多类信号（如"分析后修复"），以动作性信号为准，继续执行 0.3。

### 0.3 匹配文档，呈现选择器

1. 读项目的模块索引文件（通常为 `docs/plans/PROJECT.md`，以项目 CLAUDE.md 中的约定为准），获取全量模块清单和 spec 路径。若文件不存在或读取失败，跳至步骤 3。
2. 根据需求描述，从一级模块名称、二级模块名称、功能点描述中，找出相关性最高的 1–3 个 spec 文档。若未找到置信度足够的匹配（0 个候选，或候选与需求描述无关键词或语义重叠），跳至步骤 3。
3. **兜底扫描**（由步骤 1 文件缺失或步骤 2 匹配失败触发）：
   - 使用 Glob 工具扫描 `docs/plans/**/*.md`，排除 `PROJECT.md` 和 `init-report.md`，收集所有 spec 文件路径
   - 使用 Grep 工具搜索所有文件的 `# ` 开头行，一次操作获取全部标题作为 label
   - 将所有扫描到的 spec 全部列入选择器，由用户判断
   - 在选择器 header 中注明触发原因：步骤 1 触发时用 `"PROJECT.md 不存在，以下为目录扫描结果"`；步骤 2 触发时用 `"未在 PROJECT.md 中找到匹配模块，以下为目录扫描结果"`
4. 先调用 `ToolSearch`（query: `"select:AskUserQuestion"`）获取工具 schema，然后调用 `AskUserQuestion` 工具呈现文档选择器。

   `AskUserQuestion` 是延迟加载工具，不做 ToolSearch 直接调用会失败并退化为渲染 JSON 文本，必须先获取 schema。

   调用参数结构如下（这是参数说明，不是展示给用户的内容）：
   - `question`: `"请选择对应的技术方案文档"`
   - `header`: 正常匹配时用 `"技术文档"`；兜底扫描时按步骤 3 的触发原因填写（`"PROJECT.md 不存在，以下为目录扫描结果"` 或 `"未在 PROJECT.MD 中找到匹配模块，以下为目录扫描结果"`）
   - `multiSelect`: `false`
   - `options`: 填入匹配到的 1–3 个 spec，每项包含：
     - `label`: 模块编号/名称
     - `description`: spec 文件路径 + 功能点摘要

   - options 填入匹配到的 1–3 个 spec（label 用模块名，description 用路径 + 功能点摘要），最多 3 个，为"Other"留位；兜底扫描时列出全部扫描结果，不限 3 个
   - "Other"由工具自动提供，覆盖两种情况：手动指定路径、或说明需要新建文档

**处理用户选择：**
- 选择某个 spec → 读取对应 spec 全文 → 进入 Step 1
- 选择"Other"且用户提供了路径 → 读取该 spec → 进入 Step 1
- 选择"Other"且用户说明需要新建 → 询问模块名称和目标路径，用标准 8 节骨架创建空白 spec → 进入 Phase A

**无论哪种情况，读取 spec 后必须输出以下确认块，再进入下一步：**

```
┌─ 工作文档已建立 ──────────────────────────────────
│  路径：docs/plans/<module>/xxx-tech-spec.md
│  设计规则已加载 · 任务状态已加载 · 验收标准已加载
└───────────────────────────────────────────────────
```

## Step 1 — 判断当前阶段

**前置检查**：确认 spec 已加载（本 session 中 Step 0 已执行，或用户提供了 spec 文件路径）。若两者均未满足，输出以下提示后**立即终止，不执行后续任何步骤**：

```
⛔ 未找到工作文档。请重新运行 /spec-first 并在 Step 0 选择或创建 spec 文档。
```

**阶段判断**：扫描 spec 的任务状态（T-xxx）和验收状态（A-xxx），输出当前阶段：

| 条件 | 阶段 |
|---|---|
| 用户描述了新需求或变更 | A — 分析与更新 |
| 有"待开始"/"进行中"的 T-xxx | B — 开发 |
| 所有 T-xxx 均为 ✅ 或已取消，有"待验证"/"未通过"的 A-xxx | C — 验收 |
| 所有 T-xxx 均为 ✅ 或已取消，且所有 A-xxx ✅（A-xxx 不为空） | D — 收尾 |
| spec 无任务/验收项，或有 T-xxx 但无 A-xxx | A — 分析与更新 |

条件按表中顺序优先匹配，第 1 行优先级最高（有新需求时始终进入 Phase A）。状态混乱时询问用户确认；用户可显式指定阶段覆盖自动判断。

## Phase A — 分析与更新 spec

### A.1 确认需求理解

用 2–4 句话复述需求要点，无需等待用户回应，直接进入 A.2。（正式确认在 A.7 进行）

### A.2 识别受影响章节

列出需要改写的章节，并给出将要发生的变化：

- **已有 spec 的增量修改**：展示"修改前/修改后"对照，内容越具体越好

```
受影响章节：

■ 第 3 节（接口规范）
  变更原因：新增 firstAuthor 查询参数
  修改前：POST /item/list 接受 title、type 两个筛选条件
  修改后：POST /item/list 增加可选的 firstAuthor 筛选条件

■ 第 5 节（代码地图）
  变更原因：Mapper 方法签名增加参数
  修改前：selectPageList(String title, String type)
  修改后：selectPageList(String title, String type, String firstAuthor)
```

- **新建 spec（空白骨架）**：列出将要填入的章节和核心内容

```
将填入章节：

■ 第 1 节（需求边界）：<核心需求描述>
■ 第 3 节（接口规范）：<接口路径和方法列表>
■ 第 5 节（代码地图）：<关键文件和方法>
```

这是用户判断方案是否正确的依据，也是后续 A.4 改写时的参照。

spec 标准 8 节：① 需求边界 ② 数据库设计 ③ 接口规范 ④ 架构与设计规则 ⑤ 代码地图 ⑥ 任务状态 ⑦ 验收项 ⑧ 附录。旧版 spec 章节标题不同，按实际内容对应。

### A.3 全量证据分析（分析阶段核心步骤）

分析范围由 A.2 划定，并以用户最新补充/澄清为最高优先输入；在此基础上进行交叉核对：

- 新增/修改字段 → 读对应实体类、DTO、数据访问层映射文件
- 修改查询逻辑 → 读映射文件 + 调用该查询的 Service 方法
- 新增/修改接口 → 读 Controller/Router 类 + Service 接口定义
- 修改 DB 结构或校验现状 → 查实际表结构与关键样本数据（环境可访问时）
- 行为与实现不一致时 → 交叉读取相关日志、配置与中间层代码，定位约束来源

每读一个文件，说明发现了什么以及如何影响后续方案。例：
> "读了 `XxxMapper.xml`，`selectPageList` 已有 title/type 两个条件，新增 author 直接追加条件块，无冲突。"

### A.4 改写受影响章节（原地改写，不追加）

**核心原则：spec 记录当前状态，不记录变更历史。**

- 替换旧内容，不保留"原来是 X，现改为 Y"的说明
- 删除不再准确的字段、方法描述
- 改后重读该章节，确认与其他章节一致（DB设计 ↔ 接口规范 ↔ 代码地图）

**代码地图（第5节）原地更新规则**：修改某个方法的职责描述时，找到该方法已有的行并原地修改；**不新增同名方法行**。只有全新方法才追加行。同一张表中出现两行相同方法名是明确的错误信号。

**附录路径索引（第8节）去重规则**：修改文件路径索引时，先检查同名文件是否已有条目；若已存在，更新路径，不追加新行。

**受变更影响的任务和验收项必须重置状态：**
- 修改了某个 T-xxx 的范围（文件、方法、内容）→ 将该 T-xxx 状态重置为"待开始"
- 修改了某个 A-xxx 的断言或关联任务 → 将该 A-xxx 状态重置为"待验证"
- 未被变更触及的已通过 A-xxx，状态保持不变；已完成的 T-xxx 保留在表中，若受变更影响则重置为"待开始"

### A.5 新增 T-xxx 任务

在第六节（任务状态）追加行，每条 T-xxx 必须包含：
- 明确的文件范围（哪个文件 + 哪个方法/节点）
- 优先级：P0（阻塞）/ P1（重要）/ P2（可选）
- 状态：待开始

**任务表固定为 4 列**，表头必须是：

```
| 编号 | 内容 | 优先级 | 状态 |
```

列数一致很重要——Markdown 表格中，若某行的列数与表头不匹配，渲染器会错位显示甚至截断内容，让读者误读任务状态。优先级（P0/P1/P2）尤其容易被塞进错误的列，后续扫描任务时就会漏掉它。

✅ 好的 T-xxx：`T-11 | 在 XxxMapper.xml selectPageList 追加 author 条件；更新 XxxMapper.java 方法签名 | P0 | 待开始`
❌ 差的 T-xxx：`T-11 | 扩展筛选功能 | P0 | 待开始`（无文件无方法）
❌ 差的 T-xxx：`T-11 | ... | P0 | 待开始`（P0 没有对应的优先级列，被塞进状态列）

**代码位置定位规则**：使用方法名/类名定位，**不使用行号**（行号随代码提交立即失效）。
- ❌ `ModelCanvasServiceImpl.java:334`
- ✅ `ModelCanvasServiceImpl.getCanvasDetail()`

被取消的任务标记为"已取消"并注明原因，不删除。已完成的 T-xxx 保留在任务表中，**不在验收阶段逐条删除**——任务表将在 Phase D 所有验收通过后统一清理。编号续接规则：在当前表中（含已取消行）的最大编号基础上递增，不复用已用编号。

### A.6 新增 A-xxx 验收项

第七节（验收项）所有验收行统一使用 **A-xxx 编号，固定 4 列**：

```
| 编号 | 接口 | 验证点 | 状态 |
```

每条 A-xxx 必须包含可执行的断言——具体的 HTTP 请求 + 预期响应，或具体 SQL + 预期结果。

✅ 好的 A-xxx：`| A-05 | POST /item/list | 传 author=张三，返回 total>0，所有结果含"张三" | 待验证 |`
❌ 差的 A-xxx：`| A-05 | POST /item/list | 筛选功能正常 | 待验证 |`（无可执行断言）

追加新行前先数表头列数（4列），确保新行分隔符数量一致。列数不一致时 Markdown 会静默错位，不会报错。

**编号续接规则**：在当前表中已有 A-xxx 的最大编号基础上递增，不复用已用编号。

**安全规则**：验收章节中涉及认证信息时，**密码、token、Cookie、secret 一律用占位符**，不得明文写入 spec：
- ❌ `LOGIN_PASSWORD='Admin@123456'`
- ✅ `LOGIN_PASSWORD='<LOGIN_PASSWORD>'`
- UUID 类业务测试数据（如 modelId、userId）可以保留，但认证凭据不可明文记录。

### A.7 方案确认（确认门）

**每次进入确认门前，无论经过几轮修改，必须先完成质量检查，再输出确认材料。**

#### 质量检查（每次输出确认材料前执行）

多轮修改容易让 spec 变成局部补丁的堆叠——各章节表面上都更新了，但彼此之间开始出现裂缝。这一步的作用是在每次给用户看结果之前，以整体视角扫一遍，确保 spec 是一份内部自洽的文档，而不是几次修改的拼接产物。

重读本轮所有修改过的章节，逐项核查：

1. **三角一致性**：接口规范字段/参数 ↔ 数据库设计字段 ↔ 代码地图方法签名，三者对齐；发现不一致，以本轮用户意图为准统一修正
2. **任务可追溯**：每个 T-xxx 必须能在数据库设计（第二节）、接口规范（第三节）或代码地图（第五节）中找到对应条目；找不到则补充描述或调整任务范围
3. **断言可执行**：每个 A-xxx 的断言必须基于当前接口规范可以真实执行；若接口变了断言没更新，同步修正并重置状态
4. **全文连贯性**：通读已改章节，确认无拼接感、无遗留矛盾
5. **跨章节技术描述一致性**：若同一 SQL 片段、字段操作符或逻辑规则在多个章节均有描述（如第三章接口规范 + 第五章方法清单），必须逐字核对，以代码实现为准统一；同一技术细节不得在不同章节出现两种写法
6. **验收项 ↔ DB 字段覆盖**：若 A-xxx 断言了某个数据库字段（如 `is_deleted=true`、`category_id`），检查第二章字段说明中是否有该字段的定义；缺失则补充字段说明
7. **表格列数一致性**：扫描本轮修改过的所有表格，逐表检查每一行的列数（`|` 分隔符数量）是否与表头一致；Markdown 列数不一致时只会静默错位，不会报错，必须人工核查

检查发现问题 → 先修正，再继续。检查通过后，先展示确认材料，再调用 `ToolSearch`（query: `"select:AskUserQuestion"`）获取 schema，然后调用 `AskUserQuestion`。

#### 确认材料（必须先展示）

在调用确认器前，**重读本轮修改的章节**，确认摘要内容与文件实际一致，再向用户展示（禁止使用 `git diff` 作为唯一依据）：

- **已有 spec 的增量修改**：展示受影响章节的"修改前/修改后"对照（至少覆盖本轮改动段落）
- **新建 spec 或大范围改写**：展示"最新 spec 内容"（至少包含第 1/3/6/7 节，必要时补充其余章节）

随后补充摘要：

```
**改写章节：** X 节（需求边界）、Y 节（接口规范）

**任务变更：**
- 新增：T-12（...），T-13（...）
- 重置：T-08（原✅ → 待开始，原因：接口签名变更）
- 取消：T-06（已取消，原因：...）（如有；无则省略）
- 保持：T-01 ~ T-05, T-07

**验收项变更：**
- 新增：A-09（...）
- 重置：A-05（原✅ → 待验证，原因：断言变更）
- 保持：A-01 ~ A-04

**待确认假设：** <如有；无则省略>
```

#### AskUserQuestion 确认器

先调用 `ToolSearch`（query: `"select:AskUserQuestion"`）获取 schema，然后调用 `AskUserQuestion`。

调用参数结构如下（这是参数说明，不是展示给用户的内容）：
- `question`: `"请确认当前 spec"`
- `header`: `"Spec 确认"`
- `multiSelect`: `false`
- `options`:
  - `{ label: "确认通过，开始开发", description: "spec 已确认，进入 Phase B 按任务执行" }`
  - `{ label: "补充或澄清需求", description: "我会回到分析阶段，结合补充信息重新分析并更新 spec" }`

#### 回应处理规则

- 选"确认通过，开始开发"，且 Other 无补充 → 进入 Phase B
- 其他所有情况（选"补充或澄清需求"、或 Other 中含有任何修正/补充/澄清/"但是……"）→ 视为方案尚未确认，必须从分析阶段重新开始：
  1. 逐条理解用户反馈，复述新的需求边界与待确认点
  2. 说明将回到分析阶段重新推导方案：
     - **Why**：spec 未确认前进入实现会破坏 doc-first 前置约束——轻则大面积返工，重则错误方案固化进主干，联调阻塞、回滚成本陡增，后续任务与验收证据失去可追溯性
     - **How**：以用户最新补充/澄清为最高优先输入，全量证据分析后将结论统一写回 spec；分析阶段只读取验证，不做代码改写；完成后再次经过 A.7 确认门
  3. 回到 A.1（需求理解）→ A.2（受影响章节）→ A.3（全量证据分析）→ A.4/A.5/A.6（更新 spec）
  4. 完成后再次执行 A.7（含质量检查、确认材料展示、AskUserQuestion 确认）

**进入 Phase B 的唯一条件：用户选择"确认通过，开始开发"且没有任何保留意见。**

## Phase B — 按方案开发

> **上下文隔离策略：** B.1 制定计划在主会话执行；B.2 代码实现和 B.4 构建验证应使用 **subagent**（Agent 工具）执行，在独立上下文中运行，完成后仅返回结论和关键证据给主会话，过程日志不会污染 spec 分析所需的上下文。
>
> **使用方式：** 对每个可独立执行的任务，通过 Agent 工具创建 subagent，在独立上下文中完成代码修改、构建等操作。subagent 完成后向主会话返回结果摘要，主会话更新 spec 中的 T-xxx/A-xxx 状态。

### B.1 分析依赖，制定执行计划

读第六节，列出所有"待开始"/"进行中"的 T-xxx。**"进行中"表示上次 session 中断留下的断点标记，对应 subagent 已不存在，应视同"待开始"从头执行。**

按以下原则分组：

**并行组划分原则：**
- 同一文件的多处修改 → 串行（避免冲突）
- 不同文件且无逻辑依赖 → 并行
- P0 先于 P1 先于 P2；同优先级内再判断是否可并行

输出执行计划后等待确认再开始：
```
执行计划：
- 第 1 批（并行）：T-12（XxxServiceImpl）、T-13（XxxMapper.xml）
- 第 2 批（串行）：T-14（依赖 T-12 的结果）
```

### B.2 按批次执行

执行分两个阶段，时序如下：

**发起前**：主会话将本批所有 T-xxx 状态统一写为"进行中"（断点标记，用于 session 中断恢复），然后在单条消息中并行发起本批所有 subagent。

**收到结果后**：每个 subagent 返回后，主会话立即将该 T-xxx 状态写为 ✅ 完成（不等整批全部返回，逐条串行写入）。subagent 只执行代码，不写入 spec。若同批中某任务执行失败（编译错误、依赖缺失等普通执行错误），subagent 停止并向主会话报告失败原因，主会话定位问题后重试或按 B.3 处理（若失败原因是 spec 与代码不符）；同批中其他已完成任务保持 ✅，不整批回滚。

### B.3 发现偏差时立即暂停

若 subagent 实现中发现 spec 与实际代码不符：
1. subagent 停止并向主会话报告差异详情
2. 主会话向用户说明："spec 第 5 节写的是 X，实际代码是 Y，导致 Z 无法按方案实现"
3. 主会话提出解决方案，等用户确认后继续：
   - 若选"以代码为准更新 spec"→ 主会话原地改写 spec 对应章节（通常第 3/5 节），重新发起 subagent 按更新后的 spec 执行该任务
   - 若选"以 spec 为准修改代码"→ 重新发起 subagent 按 spec 原始方案实现，不修改 spec

### B.4 构建验证

所有 T-xxx 完成后，运行项目的构建命令验证通过。

**构建命令参考项目根目录 `CLAUDE.md` 中的"构建命令"章节。** 若 CLAUDE.md 中无构建命令，询问用户提供命令；用户确认跳过构建验证时直接进入 Phase C。

构建通过后进入 Phase C。若构建失败，subagent 定位失败原因并尝试修复后重跑；反复失败时 subagent 停止并向主会话报告，主会话向用户说明情况，等确认后继续。

## Phase C — 验收

> **上下文隔离策略：** C.2 逐条验收、C.3 根因诊断、C.4 非功能验证均使用 **subagent**（Agent 工具）执行，在独立上下文中运行。HTTP 请求/响应、SQL 查询结果等过程信息不会污染主会话上下文。

### C.0 前置检查：确认所有任务已完成

**进入验收前，必须先扫描第六节（任务状态）**，若存在状态为"待开始"或"进行中"的 T-xxx，**拒绝进入验收**，输出：

```
⛔ 以下任务尚未完成，无法进入验收阶段：
- T-xx（待开始）：<内容摘要>
- T-xx（进行中）：<内容摘要>
请先在 Phase B 完成或取消上述任务，再进入 Phase C。
```

所有 T-xxx 均为 ✅ 完成或"已取消"后，继续执行 C.1。

### C.1 准备

**具体的启动和认证命令以项目根目录 `CLAUDE.md` 中的"启动与认证"章节为准。**

通常需要：
1. 启动应用（参考 CLAUDE.md 启动命令）；若 CLAUDE.md 中无启动命令，询问用户提供；用户确认应用已在外部运行时可跳过此步直接执行 C.2
2. 获取认证信息，保存请求头（参考 CLAUDE.md 认证命令）；若 CLAUDE.md 中无认证命令，询问用户提供；用户确认无需认证时跳过此步
3. 验证 spec 第七节中的测试数据（UUID/ID/账号）是否仍有效

### C.2 逐条执行 A-xxx

**执行范围**：仅运行状态为"待验证"或"未通过"的 A-xxx，跳过已为 ✅ 的项（用于 C.3 修复后重入时避免重跑已通过测试）。

对每条待验证的 A-xxx：
1. subagent 真实调用 HTTP / 真实执行 SQL，记录证据（HTTP 状态码 + 响应摘录，或 SQL 结果），返回判定结论（pass / fail 二元）——"基本通过"无法作为交付证据
2. 主会话根据 subagent 返回结论，更新 spec 中该行状态，附证据摘录

示例：`A-05 通过 — POST /item/list {author:"张三"} → 200, total:3, 所有结果含张三`

### C.3 失败处理

有 A-xxx 未通过时，先修复再继续——下一阶段的验收建立在当前阶段已通过的基础上：

1. subagent 诊断根因（查日志、查 DB），向主会话报告根因与建议方案
2. 主会话向用户报告："A-05 未通过。原因：<根因>。建议：<方案>"
3. 主会话将对应 T-xxx 状态重置为"待开始"，回到 Phase B 修复，完成后重标 ✅；重新经过 C.0 前置检查，若应用仍在运行可跳过 C.1，重跑所有状态为"未通过"或"待验证"的 A-xxx（已 ✅ 的跳过）
4. 重验通过后，在 spec 注明"re-verified after fix"

### C.4 非功能验收

功能项全部通过后，由 subagent 验证第七节非功能验收项（启动时间、响应时间、内存占用等；构建已在 B.4 验过，此处跳过），结论返回主会话更新对应 A-xxx 状态。

## Phase D — 收尾

条件：所有 T-xxx 均为 ✅ 或已取消，且所有 A-xxx ✅

### D.1 一致性检查

逐项执行，其中 ③④ 发现问题时停止并向用户报告，等待确认后再处理；①②⑤ 为自动处理步骤：

**① 任务表清理**
扫描第六节（任务状态）并执行清理。**删除前先将所有行记入内存快照**（供 D.2 简报和 D.3 whylog 使用）：
- ✅ 完成的行 → 记入快照，然后**统一删除**
- "已取消"的行 → 记入快照，然后删除；spec 不保留已取消行
- 存在"待开始"/"进行中"行 → 不应出现（C.0 已阻止未完成任务进入验收），标记为"已取消"并在 D.2 简报"待后续迭代"节中注明，建议用户收尾后重新运行 `/spec-first` 处理

**② 验收表完整性**
扫描第七节（验收项）：
- 存在"待验证"/"未通过"行 → 报告并返回 Phase C，不进入收尾

**③ 代码与 spec 一致性**
对照第五节（代码地图）抽查本次变更的关键方法：
- 接口签名是否与第三节（接口规范）一致
- 字段/参数是否与第二节（DB 设计）一致
- 发现不一致 → 描述差异，询问用户："以代码为准更新 spec，还是以 spec 为准修改代码？"等待确认后再处理
  - 若选"以代码为准更新 spec"→ 原地改写对应章节，继续 D.1 剩余检查
  - 若选"以 spec 为准修改代码"→ 在 D.2 简报中注明该不一致项，本轮收尾后建议用户重新运行 `/spec-first` 发起新一轮分析，在 Phase A 中将修正纳入完整方案

**④ 需求边界覆盖**
对比第一节（需求边界）与本次 T-xxx 覆盖范围：
- 发现需求点无对应任务 → 在 D.2 简报中注明该遗漏项，本轮收尾后建议用户重新运行 `/spec-first` 发起新一轮分析；在第一节对该需求点标注"待后续迭代"

**⑤ 字段使用闭环**
遍历第二节所有字段，每个字段在第三节或第五节中至少有一处读写说明：
- 找不到说明的字段 → 在第二节该字段旁注明"暂未使用 / 预留"（可直接写入，无需用户确认）

所有项检查通过（或用户确认处置方案）后，进入 D.2。

### D.2 输出简报，提示收尾

整理简报所需数据来源：
- **完成任务 / 取消任务**：来自 D.1 ① 内存快照
- **验收结果**：读自 spec 第七节（A-xxx 在 D.1 阶段不删除，仍在文件中）
- **变更文件**：从任务快照的"内容"字段提取涉及的源码路径，加上 spec 文件路径

直接以消息形式发送给用户，不写入任何文件：

---

**取消任务（如有）：**
- T-xx 已取消 — <取消原因>（无则省略此节）

**待后续迭代（如有）：**
- <需求点或不一致项> — 建议下轮运行 `/spec-first` 重新分析（无则省略此节）
