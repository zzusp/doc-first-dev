# Phase B — 编排开发

**前提：Phase A 确认门已通过，需求文档中所有节已填充。**

---

## B.1 提取任务并确定批次

读取需求文档：
- 从"涉及服务与依赖顺序"节获取服务依赖顺序
- 从"任务"节按 `[服务名]` 标签分组提取各服务的 T-xxx 列表

**确定执行批次**（依赖顺序 → 串行批次，无依赖 → 并行）：

```
示例：
批次 1（并行）：data-service
批次 2（并行）：backend（依赖 data-service 完成）
批次 3（并行）：frontend（依赖 backend 完成）
```

若多个服务互不依赖，可合并到同一批次并行执行。

---

## B.2 逐批次执行

对每个批次：

**开始前**：将本批次各服务的 T-xxx 状态更新为"进行中"。

**启动 subagent**：为批次中每个服务独立启动一个 subagent，携带：
- 服务本地路径（来自 SERVICES.md）
- 需求文档路径
- 该服务的 T-xxx 列表（过滤 `[服务名]` 标签）
- 需求文档中"接口定义"节的相关内容（该服务作为调用方时需要知道提供方的接口）

subagent 职责：
1. 读取该服务的代码，理解现有结构
2. 按 T-xxx 实现代码变更
3. 每完成一个 T-xxx，在需求文档中将其标记为 ✅
4. 若发现需求文档（接口定义/代码地图）与实际代码有出入，记录差异，**不擅自决定**，在返回摘要中标注

**汇总结果**：收到 subagent 返回后，检查是否有标注的差异。

**若有 spec ↔ 代码不符**：

暂停后续批次，收集本批次所有差异，一次性展示给用户：

```
发现以下 spec 与代码不符之处，请确认如何处理：

1. backend: 需求文档定义接口返回 `{ fileUrl: string }`，
   但代码中现有同名接口返回 `{ url: string }`
   → 以 spec 为准（修改代码）/ 以代码为准（更新文档）？

2. ...
```

按用户指示逐项处理后，继续下一批次。

---

## B.3 Build 验证

所有批次完成后，按 SERVICES.md 中各服务的启动命令推断 build 命令（或询问用户），逐一执行各服务 build：

- 全部通过 → 进入 [phase-c.md](phase-c.md)
- 任意失败 → 展示错误输出，等待用户指示（修复后继续或跳过）
