# Step 0 — 需求接收

## 步骤

### 1. 接收需求

用户描述需求（口头描述、粘贴文档、或指向已有文档均可）。

### 2. 推断需求简称

根据需求内容推断一个简洁的英文文件名（kebab-case，如 `user-export`、`order-notify`）。

若不确定，向用户确认：「我将需求文档命名为 `<名称>.md`，是否合适？」

### 3. 定位需求文档

检查 `docs/plans/requirements/<名称>.md` 是否存在：

**文档已存在（续做场景）：**

读取文档，展示当前进度摘要：
```
需求「<名称>」进度：
涉及服务：<服务列表>
任务：共 N 项，已完成 M 项，待开始 K 项
验收：共 N 项，已通过 M 项，待验证 K 项
```

询问用户：
- 继续未完成部分（从哪个 Phase 继续）
- 还是本次有新的变更要追加（返回 Phase A）

**文档不存在（新需求）：**

从 [assets/multi-spec-blank.md](assets/multi-spec-blank.md) 创建 `docs/plans/requirements/<名称>.md`，填入：
- 文档标题（需求名称）
- 业务目标（基于用户描述，2-4 句，完成后让用户确认或修改）

然后进入 [phase-a.md](phase-a.md)。
