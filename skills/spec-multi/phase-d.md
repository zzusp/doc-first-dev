# Phase D — 收尾

**前提：所有 A-xxx（含 [E2E]）均为 ✅。**

---

## D.1 一致性审计

**① 接口定义 ↔ 提供方代码**

对需求文档"接口定义"节中每个提供方接口，读取对应代码的实际实现，检查：
- 方法签名（参数、返回类型）是否与文档一致
- 响应字段是否与文档一致

**② 调用方代码 ↔ 接口定义**

对需求文档"接口定义"节中每个调用方引用，读取调用方代码实际发送的请求格式，检查：
- 请求字段与提供方接口定义是否一致
- 对返回数据的处理方式是否与接口响应格式一致

**若发现不符**：

收集所有问题，一次性展示给用户：

```
发现以下不一致，请确认如何处理：

1. 接口定义：[backend] POST /api/export 响应 { fileUrl: string }
   实际代码：ExportController 返回 { url: string }
   → 以代码为准（更新需求文档）/ 以文档为准（修改代码）？
```

按用户指示更新需求文档或代码。

---

## D.2 清理任务和验收节

- **删除**：状态为 ✅ 的 T-xxx（实现已在代码中，无需留存）
- **保留**：状态为"已取消"的 T-xxx，附取消原因（作为决策记录）
- **检查**：若仍有"待开始"/"进行中"的 T-xxx，标注为风险项，提示用户处理
- **A-xxx**：保留所有 A-xxx（含通过证据），作为验收记录
- **循环计数器**：删除验收节末尾的循环计数器行（`验收循环次数：N`），过程状态交付后无意义
