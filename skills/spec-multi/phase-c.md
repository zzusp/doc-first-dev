# Phase C — 集成验收

**前提：所有服务 T-xxx 均已完成（✅），build 全部通过。**

---

## C.0 入口检查

读取需求文档"任务"节，检查是否存在状态为"待开始"或"进行中"的 T-xxx。

若有 → 拒绝进入，提示用户先完成所有任务或将未完成项标记为"已取消（附原因）"。

---

## C.1 启动所有服务

按 SERVICES.md "集成启动顺序"启动服务：

**优先**：若 workspace root 下有 `docker-compose.yml` / `docker-compose.yaml`，使用：
```
docker-compose up -d
```

**否则**：按启动顺序逐一执行各服务启动命令，每个服务启动后等待健康确认（端口响应或 health check 接口返回 200）再启动下一个。

启动后，获取需要的 auth token（参考各服务代码或 CLAUDE.md 中的 auth 说明）。

---

## C.2 执行单服务验收

**执行范围**：仅运行状态为"待验证"或"未通过"的 A-xxx（`[<服务名>]` 标签，非 `[E2E]`），跳过已为 ✅ 的项。

**验收 subagent 的信息边界（角色：验证者）：** 仅接收 A-xxx 定义、测试数据与认证信息，**不传入实现细节**（T-xxx 内容、代码地图等）。C.4 中另有**诊断 subagent（角色：排查者）**，持有完整实现上下文负责根因分析，两者职责严格区分。

对每条待验证的 A-xxx：
1. subagent 真实调用 HTTP，记录证据（状态码 + 关键字段值），返回 pass / fail 二元判定
2. 主会话执行两个写入操作：
   - **更新 A-xxx 行状态**（原地）：标记 ✅ 或 ❌
   - **追加一行到验收历史记录表**（只增不改）：若需求文档中尚无该表，在验收节末尾新建；每轮每项均追加一行，**已有行不修改**

验收历史记录表格式：

```markdown
### 验收历史记录

| 轮次 | 服务    | 项    | 结论 | 证据摘要                              | 根因（失败时）          |
|------|---------|-------|------|---------------------------------------|------------------------|
| R1   | backend | A-05  | ❌   | POST /api/export → 500                | NPE at ExportService:42 |
| R1   | backend | A-07  | ✅   | GET /api/list → 200, total:3          | —                      |
| R2   | backend | A-05  | ✅   | POST /api/export → 200, fileUrl:...   | —                      |
```

轮次 R{N} 与 C.4 步骤 3 中的循环计数器 N 保持一致；首轮记为 R1。

## C.2.5 对抗性审查

本轮实际执行的 A-xxx 全部完成后，以挑剔审查者视角扫描本轮执行结果（已跳过的 ✅ 项在首轮已审查，不需重复）：

1. 哪些 A-xxx 在实现错误时也可能侥幸通过（断言过于宽松、仅验证状态码未验证字段值）？
2. 有无边界场景（空值/越权/并发写入/重复提交）未被覆盖？

**处理规则：**
- 发现宽松断言 → 收紧对应 A-xxx 描述，重新执行该条，结果同样追加到验收历史记录表
- 发现盲区场景 → 补充新 A-xxx 并立即执行，结果同样追加到验收历史记录表
- 无问题 → 继续 C.3

---

## C.3 执行集成验收（E2E）

读取需求文档"验收"节中标签为 `[E2E]` 且状态为"待验证"的 A-xxx。

启动 subagent 执行完整跨服务请求链路（同样遵循验证者信息边界）：
- 按断言描述逐步发送请求（可能涉及多个服务的接口）
- 记录每步响应作为证据，追加到验收历史记录表
- 通过 → 标记 ✅；失败 → 标记"未通过"，记录失败在链路的哪一步

E2E A-xxx 全部执行完毕后，同样以 C.2.5 规则做对抗性审查（跨服务链路更容易有盲区，如只验了正向流程没验异常中断）。

---

## C.4 失败处理

**若有 A-xxx 标记"未通过"**：

1. 诊断 subagent（角色：排查者，持有完整实现上下文）定位根因，以**压缩摘要**（根因 + 建议，不超过 3 行）返回主会话，过程日志不带回

2. **归因判断**：主会话根据摘要判断失败根因属于哪类：
   - **实现 bug**（代码逻辑错误、字段遗漏等）→ 继续步骤 3
   - **spec/设计问题**（接口设计有缺陷、验收标准错误）→ 暂停并输出：
     ```
     ⛔ 验收失败根因属于方案层面问题，回 Phase B 修代码无法解决：
       - 问题：<具体描述>
       - 涉及服务：<服务名>
       - 涉及 A-xxx：<列表>
     请决定：
       A) 回 Phase A 重新分析
       B) 调整验收标准后继续
       C) 跳过该项，记录为待后续迭代
     ```
     等待用户选择后按指示执行，不进入后续步骤
   - **跨服务接口不匹配**（提供方与调用方接口定义不一致）→ 同上暂停，但明确标注涉及的两个服务

3. 在需求文档验收节末尾（验收历史记录表之后）更新全局循环计数器（整个需求文档维护一个，不按服务拆分）：首轮失败时新增该行并初始化 N=1，后续每轮递增 N；修复只针对失败服务，已通过服务的 ✅ A-xxx 不受影响：
   ```
   验收循环次数：N（最近一次失败：A-xx, A-xx）
   ```
   **若 N ≥ 3**，强制暂停，从验收历史记录表中提取所有 ❌ 行，输出后等待用户选择：
   ```
   ⚠️ 已进行第 N 轮验收循环，建议重新评估方案：
     <历史 ❌ 行摘要>
   请决定：
     A) 继续修复（进入第 N+1 轮）
     B) 回 Phase A 重新分析
     C) 挂起当前任务
   ```
   用户选择 A 后方可继续；N ≥ 3 的每一轮均需用户确认

4. 在需求文档"任务"节追加修复 T-xxx（含 `[服务名]` 标签）；同时扫描所有 ✅ 的 A-xxx——若其断言涉及修复 T-xxx 关联的接口或字段，一并重置为"待验证"（防止修复引入回归）
5. 回到 [phase-b.md](phase-b.md) B.2 执行修复批次：单服务 A-xxx 失败时仅修复该服务；[E2E] 失败时根据根因判断涉及的具体服务，若涉及多个服务则按依赖顺序逐服务修复。完成后回到 C.2 重新验证

6. 重验通过后，更新 A-xxx 行状态为 ✅，并在验收历史记录表追加对应的 ✅ 行

所有 A-xxx 状态均为 ✅ → 进入 [phase-d.md](phase-d.md)。
