# Phase C — 集成验收

**前提：所有服务 T-xxx 均已完成（✅），build 全部通过。**

---

## C.0 入口检查

读取需求文档"任务"节，检查是否存在状态为"待开始"或"进行中"的 T-xxx。

若有 → 拒绝进入，提示用户先完成所有任务或将未完成项标记为"已取消（附原因）"。

---

## C.1 启动所有服务

按 SERVICES.md "集成启动顺序"启动服务：

**优先**：若 workspace root 下有 `docker-compose.yml` / `docker-compose.yaml`，使用：
```
docker-compose up -d
```

**否则**：按启动顺序逐一执行各服务启动命令，每个服务启动后等待健康确认（端口响应或 health check 接口返回 200）再启动下一个。

启动后，获取需要的 auth token（参考各服务代码或 CLAUDE.md 中的 auth 说明）。

---

## C.2 执行单服务验收

读取需求文档"验收"节中标签为 `[<服务名>]`（非 `[E2E]`）且状态为"待验证"的 A-xxx。

启动 subagent 执行：
- 发送真实 HTTP 请求（按 A-xxx 中的可执行断言）
- 记录响应：状态码、关键字段值
- 将结果附在 A-xxx 后：`→ 实际: 200, fileUrl:"https://..."`
- 通过 → 标记 ✅；失败 → 标记"未通过"

---

## C.3 执行集成验收（E2E）

读取需求文档"验收"节中标签为 `[E2E]` 且状态为"待验证"的 A-xxx。

启动 subagent 执行完整跨服务请求链路：
- 按断言描述逐步发送请求（可能涉及多个服务的接口）
- 记录每步响应作为证据
- 通过 → 标记 ✅；失败 → 标记"未通过"，记录失败在链路的哪一步

---

## C.4 失败处理

**若有 A-xxx 标记"未通过"**：

1. 定位失败根因（属于哪个服务的问题）
2. 在需求文档"任务"节追加修复 T-xxx（含 `[服务名]` 标签）
3. 告知用户失败详情和追加的修复任务
4. 修复完成后回到 [phase-b.md](phase-b.md) B.2（仅执行该服务的修复批次）
5. 修复完成后回到 C.2 重新验证

---

## C.5 全部通过

所有 A-xxx 状态均为 ✅ → 进入 [phase-d.md](phase-d.md)。
