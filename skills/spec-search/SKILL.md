# /spec-search — Spec 内容快速检索

在 `docs/plans/` 下搜索技术方案文档，返回匹配结果，不修改任何文件。

**与 `/spec-first` 的区别：** `/spec-search` 只检索不修改，用于信息查找场景。若需变更文档或执行开发，应使用 `/spec-first`。

**适用场景：**
- "XX 功能在哪个模块？"
- "哪些 spec 涉及 XX 表/接口？"
- "帮我找包含 XX 关键词的技术方案"
- "项目中有哪些地方用到了 XX 字段？"

## Step 0 — 确认输入

### 0.1 解析搜索关键词

| 输入 | 行为 |
|---|---|
| 有关键词/描述 | 提取核心搜索词，进入 Step 1 |
| 无输入 | 输出提示，停止 |

无输入时提示：

```
请提供搜索关键词，例如：
  · 用户权限相关的接口
  · Kafka 消息队列
  /spec-search <关键词>
```

### 0.2 识别搜索类型

从用户描述中推断搜索意图：

| 信号 | 搜索类型 | 搜索重点 |
|---|---|---|
| 提到表名/字段名 | 数据库 | spec 第二节（数据库设计） |
| 提到接口路径/HTTP方法 | 接口 | spec 第三节（接口规范） |
| 提到文件路径/类名/方法名 | 代码 | spec 第五节（代码地图） |
| 提到任务编号（T-xxx）或状态 | 任务 | spec 第六节（任务状态） |
| 通用关键词 | 全文 | 所有章节 |

## Step 1 — 定位搜索范围

### 1.1 读取模块索引

读取 `docs/plans/README.md`（或项目 CLAUDE.md 中约定的文档目录索引），获取全量模块清单和 spec 路径。

若索引文件不存在：

```
⚠️ 未找到文档索引（docs/plans/README.md）
请确认已运行 /spec-init 初始化。
```

### 1.2 确定搜索目标

根据 Step 0 推断的搜索类型，确定需要读取的章节范围，避免读取无关内容。

---

## Step 2 — 执行搜索

### 2.1 搜索策略

按以下优先级搜索：

1. **模块索引优先**：先在 `docs/plans/README.md` 的功能清单总表中搜索，匹配一级/二级模块名和功能点描述
2. **Spec 内容搜索**：使用 Grep 工具在 `docs/plans/` 目录下搜索关键词，按搜索类型限定文件范围
3. **结果排序**：按相关度排序——标题匹配 > 章节标题匹配 > 正文匹配

### 2.2 搜索执行

使用 Grep 工具在 `docs/plans/` 下搜索关键词，读取匹配行的上下文（前后各 2–3 行），提取有意义的摘要。

---

## Step 3 — 输出结果

### 3.1 格式化输出

以结构化格式展示搜索结果：

```
┌─ Spec 搜索结果 ─────────────────────────────────
│  关键词：<搜索词>
│  搜索范围：docs/plans/（<N> 个模块）
│  匹配结果：<M> 处
├────────────────────────────────────────────────
│
│  ■ <模块名> — <spec 路径>
│    匹配：<匹配的章节> · <上下文摘要>
│
│  ■ <模块名> — <spec 路径>
│    匹配：<匹配的章节> · <上下文摘要>
│    匹配：<另一处匹配>
│
│  ■ 未匹配的模块：<模块C>、<模块D>
│
└────────────────────────────────────────────────
```

### 3.2 无结果时

```
未在 docs/plans/ 下找到与 "<关键词>" 匹配的内容。

建议：
  · 检查关键词拼写或使用更通用的词
  · 若功能尚未记录在 spec 中，可运行 /spec-init 初始化
  · 若需要跨项目搜索，请直接在项目源码目录中使用 Grep 工具
```

### 3.3 搜索完成后停止

不自动执行任何修改操作。若用户基于搜索结果想要执行开发任务，提示使用 `/spec-first <需求描述>`。
