# /spec-analyze — 只读分析

对项目代码和 spec 进行只读分析，输出分析结果和建议，**不修改任何文件**。

**与 `/spec-first` 的区别：**

| 维度 | `/spec-analyze` | `/spec-first` |
|------|-----------------|---------|
| 目的 | 只读分析，输出结论 | 文档驱动开发，修改文件 |
| 文件操作 | 仅读取 | 读取 + 修改 |
| 适用场景 | 了解现状、审查、诊断 | 需求变更、Bug修复、新功能 |

**适用场景：**
- "帮我分析一下这个接口的实现逻辑"
- "这个模块有什么潜在问题？"
- "XX 变更会影响哪些模块？"
- "帮我审查一下这段代码"
- "这个 Bug 可能的原因是什么？"

## Step 0 — 接收分析请求

### 0.1 确认输入

检查用户是否提供了具体的分析对象或问题：

**无输入时**，输出以下提示，停止：

```
请描述您的分析需求，例如：
  · 分析用户管理模块的代码质量
  · XX 接口的性能瓶颈在哪里？
  · 这个变更会影响哪些模块？
  · 帮我审查这个 Service 的实现逻辑
```

**有输入时**，继续。

### 0.2 确认为只读分析

判断用户意图是否为只读分析。以下信号表明是只读分析：

- "分析"、"查看"、"审查"、"诊断"、"评估"、"了解"
- "有哪些问题"、"影响范围"、"为什么"
- 不包含"修改"、"修复"、"实现"、"新增"等动作词

**若用户意图实际是执行开发任务**，提示用户使用 `/spec-first`：

```
此需求涉及代码修改，建议使用 /spec-first <需求描述> 进入文档驱动开发流程。
```

用户确认仍要只读分析时，继续。

## Step 1 — 确定分析范围

### 1.1 定位相关文档

读取 `docs/plans/README.md`（或项目约定的文档索引），找到与用户需求相关的 spec 文档。

若存在相关 spec，读取全文作为分析的基准上下文。

### 1.2 定位相关代码

基于用户描述和 spec 中的代码地图（第五节），确定需要读取的源码文件范围。

**范围控制原则：**
- 从 spec 的代码地图出发，仅读取明确相关的文件
- 分析过程中如需扩大范围，说明原因后再读取
- 避免无目标的文件扫描

---

## Step 2 — 执行分析

### 2.1 分析框架

根据用户的分析需求，选择合适的分析维度：

| 需求类型 | 分析维度 |
|---|---|
| 代码审查 | 命名、结构、异常处理、边界条件、可维护性 |
| 影响分析 | 调用链、数据流、依赖关系、受影响模块 |
| 问题诊断 | 日志、错误信息、数据状态、代码路径 |
| 方案评估 | 可行性、风险、工作量、替代方案 |
| 现状了解 | 模块职责、核心逻辑、数据结构、接口行为 |

### 2.2 证据收集

每读取一个文件，记录：
- 发现了什么
- 与用户问题的关联
- 是否影响分析结论

### 2.3 交叉验证

将代码事实与 spec 描述进行交叉比对（若 spec 存在）：

- Spec 描述的行为与代码实现是否一致
- Spec 中的接口/字段与代码是否对齐
- 发现的不一致作为分析结论的一部分输出

---

## Step 3 — 输出结论

### 3.1 结构化输出

```
┌─ 分析结论 ─────────────────────────────────────
│  分析对象：<模块名/接口名/文件路径>
│  分析类型：<代码审查/影响分析/问题诊断/方案评估>
├────────────────────────────────────────────────
│
│  ■ 分析摘要
│    <2-3 句话概括核心发现>
│
│  ■ 详细发现
│    1. <发现1>
│       位置：<文件:行号>
│       说明：...
│    2. <发现2>
│       ...
│
│  ■ Spec 一致性（如适用）
│    ✅ 一致：<项目>
│    ⚠️ 不一致：<项目>（spec 描述为 X，代码实际为 Y）
│
│  ■ 建议（如有）
│    - <建议1>
│    - <建议2>
│
└────────────────────────────────────────────────
```

### 3.2 后续引导

分析完成后，根据结果引导用户：

- 若发现问题需修复 → 提示使用 `/spec-first <问题描述>` 进入开发流程
- 若发现 spec 与代码不一致 → 提示使用 `/spec-first <描述>` 更新 spec
- 若仅是了解性分析 → 结束，无后续操作
- 若需要更深入的分析 → 提示可以针对具体子问题继续分析

### 3.3 重要约束

**本 skill 执行全程不修改任何文件。** 若分析过程中发现需要修改，在结论中说明，由用户决定是否使用 `/spec-first` 执行。
