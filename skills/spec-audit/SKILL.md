# /spec-audit — Spec 全局健康度审计

对 `docs/plans/` 下所有模块的 spec 执行批量结构健康度扫描，快速识别需要关注的问题模块。

**与 `/spec-check` 的区别：**

| 维度 | `/spec-check` | `/spec-audit` |
|------|---------------|---------------|
| 范围 | 单模块深度检查 | 全局批量扫描 |
| 关注点 | 接口↔DB↔代码一致性 | 结构健康度 |
| 深度 | 逐字段/方法验证 | 快速打标签 |
| 用途 | 深入诊断 | 日常巡检 |

## Step 0 — 收集模块列表

### 0.1 读取模块索引

读取 `docs/plans/PROJECT.md`，获取全量模块清单和 spec 路径。

若 PROJECT.md 不存在，输出提示后停止：

```
⚠️ 未找到 docs/plans/PROJECT.md
请确认 /spec-init 已运行，或手动创建模块索引。
```

### 0.2 定位所有 Spec

对每个模块，定位其 spec 文件（通常为 `docs/plans/<module>/<module>-tech-spec.md`）。

输出扫描范围：

```
┌─ Spec 审计范围 ──────────────────────────────────
│  模块总数：<N>
│  扫描路径：docs/plans/
└───────────────────────────────────────────────────
```

## Step 1 — 结构健康度检查

对每个 spec 执行以下检查项，输出检查结果。

### 1.1 空章节检测

检查 8 节骨架中是否有空章节（仅有占位符或 `<!-- -->` 注释而无实质内容）。

**判定规则：**
- 章节内容仅包含 `<!-- ... -->` 或 `<!---->` 注释
- 章节内容仅包含 `> 待补充` / `> 待填写` / `> ...`
- 章节内容为空或仅含空白字符

**分级：**
- 🔴 严重：需求边界（第一节）为空
- 🟡 警告：其他章节为空

### 1.2 孤立任务检测

检查第六节（任务状态）中的每个 T-xxx：

- T-xxx 描述是否模糊（无文件路径、无方法名）
- T-xxx 描述中引用的文件/方法是否能对应到第五节（代码地图）
- 长时间未更新的任务（可选，结合 git 时间判断）

**分级：**
- 🟡 警告：描述模糊（但有明确文件范围）
- 🔴 严重：无法追溯到代码地图中的任何条目

### 1.3 悬空验收检测

检查第七节（验收项）中的每个 A-xxx：

- A-xxx 是否关联了有效的 T-xxx
- 关联的 T-xxx 是否存在
- A-xxx 的断言是否为空或占位符

**分级：**
- 🟡 警告：断言为占位符
- 🔴 严重：A-xxx 关联了不存在的 T-xxx

### 1.4 状态不一致检测

检查 T-xxx 与 A-xxx 之间的状态逻辑：

| 场景 | 判定 | 严重性 |
|------|------|--------|
| T-xxx ✅ 完成，但关联的 A-xxx 全部仍为"待验证" | ⚠️ 矛盾 | 🟡 警告 |
| 所有 A-xxx ✅ 通过，但关联的 T-xxx 仍有未完成 | ⚠️ 矛盾 | 🔴 严重 |
| T-xxx 标记"已取消"，但关联的 A-xxx 仍为"待验证" | ⚠️ 不同步 | 🟡 警告 |
| Phase 1 已开始，但 Phase 0 仍有待验证项 | ⚠️ 依赖缺失 | 🟡 警告 |

### 1.5 章节关联性检测

检查各节之间的引用是否一致：

- 第三节（接口规范）中引用的字段，是否在第二节（数据库设计）中有对应说明
- 第五节（代码地图）中引用的文件路径，是否在项目结构中存在
- 第六节和第七节的 T-xxx / A-xxx 编号是否连续

**分级：**
- 🟡 警告：引用不完整
- 🔴 严重：引用目标不存在

## Step 2 — 输出审计汇总报告

### 2.1 模块级报告

对每个模块输出：

```
■ <模块名>
  路径：docs/plans/<module>/xxx-tech-spec.md
  问题数：<N>（🔴 <严重数> / 🟡 <警告数>）
  ├─ 🔴 严重：<问题1>
  ├─ 🔴 严重：<问题2>
  └─ 🟡 警告：<问题3>
```

### 2.2 全局汇总

```
┌─ Spec 健康度审计报告 ─────────────────────────────
│  审计时间：<当前时间>
│  扫描模块：<N> 个
├───────────────────────────────────────────────────
│
│  ■ 健康度分布
│    🟢 健康（无问题）：<N> 个
│    🟡 警告（1-2个问题）：<N> 个
│    🔴 严重（3+问题）：<N> 个
│
│  ■ 问题类型统计
│    空章节：<N> 处
│    孤立任务：<N> 处
│    悬空验收：<N> 处
│    状态不一致：<N> 处
│    章节关联性问题：<N> 处
│
│  ■ 需要关注的模块
│    <模块A> — 🔴 严重问题 × <N>
│    <模块B> — 🟡 警告 × <N>
│
└───────────────────────────────────────────────────
```

### 2.3 健康度评分（可选）

计算整体健康度评分（0-100）：

```
健康度评分：<分数>
  🟢 优秀（90+）：无空章节、无严重问题
  🟡 良好（70-89）：有警告项，无严重问题
  🔴 需改进（<70）：存在严重问题
```

## Step 3 — 建议操作

根据审计结果输出建议：

| 结果 | 建议 |
|------|------|
| 全部 🟢 健康 | 输出"spec 全局健康度良好，继续保持" |
| 有 🟡 警告 | 建议逐个模块修正，可使用 `/spec-first <描述>` 补充空章节 |
| 有 🔴 严重 | 建议优先修正严重问题，防止文档腐化影响开发 |
| 大量空章节 | 建议运行 `/spec-init` 重新生成基线 |

## Step 4 — 生成仪表盘数据

审计完成后，将 JSON 数据写入 `docs/plans/dashboard-data.json`：

```
┌─ Spec 仪表盘数据已生成 ───────────────────────
│  路径：docs/plans/dashboard-data.json
│  审计时间：<timestamp>
│  健康度评分：<score>
│
│  配合 /spec-dashboard 生成 HTML 仪表盘。
└──────────────────────────────────────────────
```

JSON 数据格式：

```json
{
  "timestamp": "<ISO时间戳>",
  "projectRoot": "<项目根目录>",
  "score": <健康度评分>,
  "totalModules": <模块总数>,
  "healthyModules": <健康模块数>,
  "warningModules": <警告模块数>,
  "criticalModules": <严重模块数>,
  "pendingTasks": <待完成任务数>,
  "pendingAssertions": <待验收项数>,
  "modules": [
    {
      "name": "<模块名>",
      "path": "docs/plans/<module>/<module>-tech-spec.md",
      "status": "healthy | warning | critical",
      "taskTotal": <任务总数>,
      "taskDone": <已完成任务数>,
      "assertTotal": <验收项总数>,
      "assertDone": <已验收项数>,
      "issueCount": <问题数>
    }
  ],
  "issues": [
    {
      "type": "empty | orphan | dangling | inconsistent",
      "module": "<模块名>",
      "description": "<问题描述>"
    }
  ]
}
```

配合 `/spec-dashboard` 生成 HTML 仪表盘。

## 与 `/spec-check` 的配合使用

```
日常巡检 → /spec-audit（快速全局扫一遍）
        ↓ 发现问题模块
深入诊断 → /spec-check <模块名>（逐字段验证）
        ↓ 确认问题
修正更新 → /spec-first <需求>（更新 spec）
```
