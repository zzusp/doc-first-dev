# /spec-init — 已有项目初始化

将已有代码项目的现状逆向生成为 doc-first 技术方案基线。

**前置条件：**
- 项目已有可运行的代码
- 未初始化过 doc-first（即 `docs/plans/` 下无有效模块 spec）

**输出产物：**
- `docs/plans/README.md` — 模块索引 + 功能清单（用户确认后生成）
- `docs/plans/<module>/xxx-tech-spec.md` × N — 各模块基线 spec
- `docs/plans/init-report.md` — 初始化报告

## Step 1 — 分析项目

### 1.1 检测项目类型

扫描项目根目录，识别语言/框架：

| 信号 | 推断 |
|---|---|
| `pom.xml` / `build.gradle` | Java/Maven 或 Gradle |
| `package.json` | Node.js / TypeScript |
| `go.mod` | Go |
| `Cargo.toml` | Rust |
| `requirements.txt` / `pyproject.toml` | Python |
| `*.csproj` | C# / .NET |

记录项目类型，后续分析时针对该语言的代码结构特点调整。

### 1.2 扫描模块结构

遍历 `src/` 下的一级目录，推断一级模块。

每个一级模块下继续扫描二级目录，推断二级模块。

**输出：模块树草稿**

```
一级模块：src/main/java/com/example/user
  二级模块：controller / service / mapper / entity
```

### 1.3 提取接口与功能映射

根据语言/框架，扫描对应文件。详见 [reference/languages.md](reference/languages.md)。

通用规则（所有语言）：

| 文件类型 | 扫描内容 |
|---|---|
| Controller / Router | 接口路径、HTTP 方法、参数名 |
| Service 接口 | 业务方法签名 |
| Model / Entity / DTO | 字段名、类型、注解约束 |

### 1.4 推断功能点

从接口和方法名推断功能点：

- 给有意义的名字：❌ `selectPageList` → ✅ `用户分页查询`
- 同一 Controller 的多个方法若是同一资源的 CRUD，合并为一个功能点
- 跨 Controller 的相似命名若是同一功能的不同入口，合并

### 1.5 数据库扫描（可选）

**询问用户：**

```
本项目是否需要扫描数据库表结构？
如需要，提供数据库连接信息（如：mysql -h localhost -u root -p dbname）。
输入"跳过"则不扫描数据库。
```

跳过：仅基于代码分析生成功能清单。
提供连接信息：执行查询获取表结构，用于 spec 的"数据库设计"章节。

### 1.6 扫描已有文档

扫描 `docs/` 下所有 `.md` 文件，记录已有文档路径和标题。

用于 Step 3 判断哪些模块已有文档需要迁移。

### 1.7 生成功能清单草稿

基于以上分析，生成 markdown 格式的功能清单：

```
## 功能清单草稿

> ⚠️ 以下为工具推断结果，请逐项确认。低置信度项已标记。

### 一级模块：<模块名>
> 路径：<src 下的目录路径>

| 二级模块 | 功能点 | 相关接口 | 置信度 | 备注 |
|---|---|---|---|---|
| 用户CRUD | 用户分页查询 | POST /user/list | 高 | |
| | 用户详情 | GET /user/{id} | 高 | |
| | ⚠️ 删除用户 | DELETE /user/{id} | ⚠️ 低 | 仅发现路由，无对应Service方法 |
```

**置信度标注规则：**
- 高：接口、方法、表结构均有明确代码证据
- ⚠️ 低：仅有部分证据，需要人工确认

**输出后暂停，等待用户进入 Step 2。**

## Step 2 — 确认功能清单

展示 Step 1 的结果，请用户确认或修正：

```
┌─ 功能清单确认 ──────────────────────────────────────
│
│  分析完成，共发现 <N> 个模块，<M> 个功能点
│  ⚠️ 标记了 <X> 个低置信度项，请在确认前逐项检查
│
│  ▼ 一级模块：<模块名>
│    ▼ 二级模块：<子模块>
│      ✅ <功能点>（高置信）
│      ⚠️ <功能点>（低置信，请确认）
│
│  ▼ 一级模块：<模块名>
│    ...
│
│  已有文档：<M> 个模块已有技术文档（Step 3 将处理）
│
└─────────────────────────────────────────────────────
```

### 2.1 用户修正

请用户在对话中说明需要修正的内容，支持以下操作：

- **重命名**：某个模块或功能点的名称不准确
- **合并**：两个功能点实际是同一个
- **拆分**：某个功能点包含多个独立功能
- **移动**：某个功能点归属的模块不对
- **删除**：某个功能点是误判
- **新增**：漏掉了某个功能点

收到用户修正后，更新清单，直到用户表示"确认清单准确"。

### 2.2 写入 README.md

用户确认后，将最终清单写入 `docs/plans/README.md`：

```
# docs/plans — 功能模块技术方案

本目录存放项目各功能模块的完整技术方案文档，同时维护全量功能清单（见下方功能清单总表）。

---

## 功能清单总表

| 一级模块 | 二级模块 | 功能点 | 技术方案 |
|---|---|---|---|
| **1. <模块名>** | <子模块> | <功能点> | — |
```

> 表格中的"技术方案"列在 Step 3 生成 spec 后补充链接。

## Step 3 — 生成 Spec 基线

基于确认后的 `docs/plans/README.md`，按模块逐个生成基线 spec。

### 3.1 模块分类

对每个模块，判断其已有文档状态：

| 状态 | 处理方式 |
|---|---|
| `docs/plans/` 下有 doc-first 格式的 spec | ✅ 直接使用，跳过生成 |
| `docs/` 下有非标准技术文档 | 触发迁移选择（见下方） |
| 无任何文档 | 逆向生成基线 spec |

### 3.2 非标准文档迁移

对有非标准文档的模块，输出迁移选择：

```
┌─ <模块名> 已有文档迁移 ────────────────────────────
│  发现：docs/xxx.md
│  内容：接口说明 × 5，表结构 × 3
├───────────────────────────────────────────────────
│  [1] 转为 doc-first 格式
│      → 映射到 8 节结构，保留现有内容
│
│  [2] 与代码对比后合并
│      → 生成逆向版本 → 展示差异 → 用户选择保留哪个
│
│  [3] 保留原文档，新建空白 spec
│      → 原文档移至 docs/archives/
│
│  [4] 跳过该模块
└───────────────────────────────────────────────────
```

调用 `AskUserQuestion` 等待用户选择。

### 3.3 逆向生成基线 Spec

对无文档的模块，逆向生成 spec。生成原则：

**第一节（需求边界）：**
- 1.1 业务背景：基于模块/类名推断，留"待补充"占位
- 1.2 核心需求：来自功能清单（Step 2 确认版），包含接口和数据表引用
- 1.3 明确排除范围：留空，标注"待补充"

**第二节（数据库设计）：**
- 仅描述代码中有实际映射的表和字段
- 无代码证据的约束、索引不写入
- 无 DB 扫描时注明"待补充（需要 DB 连接信息）"

**第三节（接口规范）：**
- 仅列出代码中实际存在的接口
- 不推测行为、筛选参数、响应结构（除非有明确代码证据）
- 格式：表格式，Path + Method + 功能名，不写详细参数

**第四节（架构与设计规则）：**
- 仅写代码注释或常量中明确写出的规则
- 不推测架构模式、数据流向

**第五节（代码地图）：**
- 列出关键文件路径（Controller、Service、Mapper、Entity、DTO）
- 公开方法签名 + 职责推断
- 不写私有方法细节

**第六节（任务状态）：**
- 一条汇总任务：`T-01 | 建立初始基线文档 | — | ✅ 完成`
- 标注基线版本：`对应代码版本：<git commit hash>`

**第七节（验收项）：**
- 基线验收项：标注"现状通过"，不写具体 HTTP 断言
- 非功能项：构建 ✅、启动 ✅

**第八节（附录）：**
- 按需填写，不强制
- 枚举值仅填有明确代码证据的

### 3.4 写入文件

每个模块的 spec 写入 `docs/plans/<module>/xxx-tech-spec.md`。

**复制脚本到项目目录：**

```bash
mkdir -p .claude/hooks
cp ~/.claude/skills/spec-init/assets/generate-baseline-specs.py .claude/hooks/
```

**执行写入：**

```bash
python3 .claude/hooks/generate-baseline-specs.py  # 或 python .claude/hooks/generate-baseline-specs.py
```

### 3.5 更新 README.md

spec 生成后，在 README.md 的功能清单总表中补充技术方案列的链接：

```
| **1. 用户管理** | 用户CRUD | 用户分页查询 | [用户管理技术方案](user/user-tech-spec.md) |
```

## Step 4 — 输出初始化报告

将以下内容写入 `docs/plans/init-report.md`：

```
# 初始化报告

生成时间：<当前时间>
对应代码版本：<git commit hash>

## 处理摘要

| 类别 | 数量 |
|---|---|
| 直接使用（已有doc-first标准） | N |
| 迁移转换（非标准→doc-first） | N |
| 逆向生成（无文档） | N |
| 跳过（待手动处理） | N |

## 模块列表

| 模块 | 类型 | Spec路径 |
|---|---|---|
| 用户管理 | 逆向生成 | docs/plans/user/user-tech-spec.md |
| 订单管理 | 迁移转换 | docs/plans/order/order-tech-spec.md |

## 冲突记录（如有）

| 模块 | 位置 | 问题 | 处理 |
|---|---|---|---|
| 订单管理 | 接口规范 | 文档标注支持 status 筛选，代码未实现 | ⚠️ 已标注存疑 |

---

## 下一步

1. 运行 `git add . && git commit -m "初始化 doc-first 技术方案基线"` 提交本次产物
2. 从现在开始，任何需求变更、Bug修复、新功能，都使用 `/spec-first <需求描述>` 驱动
3. 如发现 spec 与代码不一致，使用 `/spec-first` 修正 spec
```

```
git add .
git commit -m "初始化 doc-first 技术方案基线"

此后，按 `/spec-first <需求描述>` 流程推进后续需求。
```
