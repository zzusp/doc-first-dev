## 技术方案文档标准

每份技术方案使用以下 8 节标准结构（章节标题可按模块特点微调）：

| 章节 | 职责 |
|---|---|
| 一、需求边界 | 核心功能清单 + 明确排除范围 |
| 二、数据库设计 | 表清单、字段语义、JSONB 内部结构、约束规则 |
| 三、接口规范 | 接口列表 + 关键行为规则 |
| 四、架构与设计规则 | 模块关系、数据流、**设计规则**（以"规则：xxx"形式明确列出） |
| 五、代码地图 | 关键文件路径 + 方法职责表 |
| 六、任务状态 | T-xxx 任务表，实时更新 |
| 七、验收项 | 验收前准备 + A-xxx 验收表，断言要求可执行可观测 |
| 八、附录 | 枚举值、常量、固定测试数据 |

新建 spec 时使用 `~/.claude/skills/spec-init/assets/tech-spec-blank.md` 骨架，或运行 `/spec <需求描述>` 选择"新建文档"自动生成。

### T-xxx 任务格式

```
| T-xxx | <具体文件 + 方法/XML节点> | P0/P1/P2 | 待开始/进行中/✅ 完成/已取消（原因） |
```

- P0 阻塞、P1 重要、P2 可选
- 取消时保留行并注明原因，不删除

✅ `T-11 | 在 XxxMapper.xml selectPageList 追加 author 条件；更新 XxxMapper.java 方法签名 | P0 | 待开始`
❌ `T-11 | 扩展筛选功能 | P0 | 待开始`（无文件无方法）

### A-xxx 验收格式

```
| A-xxx | <验收项名称> | T-xxx | <具体 HTTP 请求 + 预期响应，或 SQL + 预期结果> | 待验证/✅ 通过/未通过 |
```

✅ `A-05 | author 筛选 | T-11 | POST /item/list 传 author=张三，返回 total>0，所有结果含"张三" | 待验证`
❌ `A-05 | 筛选功能正常 | T-11 | 功能正确 | 待验证`（无法执行）

## 文档维护原则

### 文档先行

**任何需求调整、功能变更、新增任务，都必须先更新文档，再开始开发。**

运行 `/spec <需求描述>` 驱动完整开发周期（分析 → 更新 spec → 开发 → 验收 → 收尾）。

### 更新前先分析

更新 spec 前，靶向探索受影响的代码：

- 修改字段 → 读实体类、DTO、数据访问层
- 修改查询 → 读数据访问层 + 调用方 Service
- 新增接口 → 读 Controller/Router + Service 接口定义
- 修改 DB → 查实际表结构

### 覆盖，不追加

spec 记录**当前状态**，不记录变更历史。字段改名、行为变化、断言变化，直接改旧内容。

### 代码与文档同提交

spec 更新和代码变更放在同一次 git commit 中。

### 文档与代码不符时问用户

发现 spec 与实际代码不一致，不自行判断，向用户指出差异，等确认后更新文档并继续。
