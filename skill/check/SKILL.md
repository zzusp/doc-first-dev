# /spec-check — Spec 健康检查

对技术方案文档执行按需健康检查，不修改任何文件。

**适用场景：**
- 开发前想确认 spec 是否可靠
- 交付前做最终验证
- 日常巡检，确保 spec 与代码未 drifted

## Step 0 — 确定检查目标

### 0.1 解析输入

| 输入 | 行为 |
|---|---|
| 无参数 | 读取 `docs/plans/README.md`，展示模块选择器 |
| 模块名或 spec 路径 | 直接定位到对应 spec |
| `--all` | 检查 README.md 中所有模块 |

### 0.2 定位 spec 文件

1. 读取项目的 `docs/plans/README.md`，获取全量模块清单。
2. 根据输入匹配到目标模块，定位 spec 路径（通常为 `docs/plans/<module>/<module>-tech-spec.md`）。
3. 若 spec 文件不存在，输出提示后停止：

```
⚠️ 未找到 spec 文件：<路径>
请确认模块名正确，或先运行 /spec-init 初始化。
```

4. 读取目标 spec 全文，进入 Step 1。

无参数时，调用 `AskUserQuestion` 展示选择器：

```json
{
  "questions": [{
    "question": "请选择要检查的模块",
    "header": "Spec 检查",
    "multiSelect": false,
    "options": [
      { "label": "<模块名>", "description": "<spec路径>" }
    ]
  }]
}
```

## Step 1 — Spec 内部一致性检查

读取 spec 全文后，逐项执行以下检查。每项给出 ✅ 或 ⚠️ 判定。

### 1.1 三角一致性

检查接口规范（第三节）字段/参数 ↔ 数据库设计（第二节）字段 ↔ 代码地图（第五节）方法签名，三者是否对齐。

具体检查：
- 接口入参中引用的字段，在数据库设计中有对应说明
- 代码地图中的方法参数，与接口规范中的参数一致
- 数据库设计中的字段，在代码地图中有对应的访问方法

发现不一致时记录：`⚠️ <字段/参数名> 在 <章节> 中为 X，在 <章节> 中为 Y`

### 1.2 任务可追溯

检查第六节每个 T-xxx：
- 描述中引用的文件/方法，是否在第三节（接口规范）或第五节（代码地图）中能找到对应条目
- 无法追溯的任务标记为孤立任务

### 1.3 断言可执行

检查第七节每个 A-xxx：
- 断言中引用的接口路径和 HTTP 方法，是否在第三节（接口规范）中存在
- 断言中引用的参数/字段，是否在接口规范中有定义
- 不可执行的断言标记出来

### 1.4 状态完整性

检查第六节和第七节：
- 每个 T-xxx 和 A-xxx 是否有明确状态（非空、非占位符）
- 状态值是否为合法枚举：待开始 / 进行中 / ✅ 完成 / 已取消（T-xxx），待验证 / ✅ 通过 / 未通过（A-xxx）

### 1.5 状态逻辑

检查 T-xxx 与 A-xxx 之间的状态逻辑是否自洽：
- 所有 A-xxx ✅ 通过，但关联的 T-xxx 仍有未完成 → ⚠️ 矛盾
- T-xxx 已 ✅ 完成，但关联的 A-xxx 全部仍为"待验证" → ⚠️ 可能遗漏验收
- T-xxx 标记"已取消"，但关联的 A-xxx 仍为"待验证"而非同步取消 → ⚠️

### 1.6 章节完整性

检查标准 8 节骨架是否齐全：需求边界、数据库设计、接口规范、架构与设计规则、代码地图、任务状态、验收项、附录。

标记空章节（仅有占位符或 `<!-- -->` 注释而无实质内容），但不视为错误——仅提示待补充。

## Step 2 — Spec 与代码符合度检查

基于 spec 第五节（代码地图）中列出的文件路径，读取对应源码文件，交叉比对。

**原则：仅读取 spec 明确引用的文件，不扩散扫描。**

### 2.1 接口存在性

检查第三节列出的每个接口（Method + Path）：
- 在代码地图中引用的 Controller/Router 文件中搜索该路径和方法
- 存在 → ✅
- 不存在 → ⚠️，记录 spec 中有但代码中未找到的接口

### 2.2 方法存在性

检查第五节列出的每个方法签名：
- 在代码地图中引用的文件中搜索该方法
- 方法名存在但签名不符 → ⚠️，展示 spec 签名 vs 代码实际签名
- 方法名不存在 → ⚠️

### 2.3 字段存在性

检查第二节列出的每个字段：
- 在 Entity/Model 文件中搜索该字段
- 存在 → ✅
- 不存在 → ⚠️，记录 spec 中有但代码中未找到的字段

### 2.4 文件可达性

检查第五节列出的每个文件路径是否真实存在。不存在的文件标记为 ⚠️。

## Step 3 — 输出检查报告

以结构化格式直接展示，不写入文件：

```
┌─ Spec 健康检查报告 ────────────────────────────────
│  模块：<模块名>
│  Spec：<spec 相对路径>
│  检查时间：<当前时间>
├───────────────────────────────────────────────────
│
│  ■ 内部一致性
│    <检查项>：<判定> <详情>
│    ...
│
│  ■ 代码符合度
│    <检查项>：<判定> <详情>
│    ...
│
│  ■ 总结
│    ⚠️ 项：<N> 个
│    ❌ 严重问题：<M> 个
│    建议：<操作建议>
│
└───────────────────────────────────────────────────
```

`--all` 模式下，先逐模块检查，最后输出汇总：

```
┌─ 全局检查汇总 ──────────────────────────────────────
│  检查模块数：<N>
│  健康（全部 ✅）：<M> 个
│  有问题（有 ⚠️）：<K> 个
│
│  模块明细：
│    <模块A> — ✅ 健康
│    <模块B> — ⚠️ 3 个问题（详情见上方报告）
│    ...
└───────────────────────────────────────────────────
```

## Step 4 — 建议操作

根据检查结果：

| 结果 | 建议 |
|---|---|
| 全部 ✅ | 输出"spec 健康检查通过"，无后续操作 |
| 有 ⚠️ 但无 ❌ | 建议使用 `/spec <需求>` 修正不一致项，修正后可再次 `/spec-check` 验证 |
| 有 ❌ 严重问题 | 明确标注问题严重性，建议优先修正后再继续开发 |
| spec 文件缺失或格式异常 | 建议运行 `/spec-init` 初始化或手动创建 |

检查完成后停止，不自动执行修复。
